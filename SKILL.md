# 微信自动化技能

## 1. Description（技能详细说明）

这是一个微信客户端自动化控制工具，可以唤醒微信窗口、截图、识别界面内容、模拟点击和输入操作、发送消息、检查新消息。

**注意：微信需要用户手动打开，工具不会自动启动微信。**

**核心特性：**
- 自动唤醒微信窗口（Ctrl+Alt+W 快捷键，仅当微信在托盘区时）
- 所有操作前自动激活窗口，确保窗口在前台
- 每次操作都获取最新截图，不使用缓存
- 支持多窗口标题匹配（"微信"、"WeChat"、"Weixin"）

## 2. When to use（触发场景）

当用户说以下内容时，使用此技能：

- "打开微信"、"唤醒微信"、"显示微信"
- "帮我截个微信的图"、"微信截图"
- "微信界面上有什么内容"、"识别微信界面文字"
- "点击微信上的XXX"、"在微信里输入XXX"
- "发送消息"、"在当前聊天发送XXX"
- "滚动微信聊天记录"
- "帮我操作微信"相关任务
- "微信状态"、"微信是否在运行"
- **"看看谁给我发消息了"、"查看聊天列表"、"检查新消息"** → 使用 `get_chat_list` 或 `check_new_messages`
- **"查看和XXX的聊天记录"、"看看和XXX的聊天"、"读取XXX的聊天内容"** → 使用完整流程（见第5.1节）
- **"给XXX发消息"、"和XXX说话"、"告诉XXX"** → 使用完整流程（见第5.2节）
- **"回复XXX"** → 使用 `auto_reply`

## 3. How to use（调用逻辑）

**调用格式：**
```bash
python {baseDir}/agent.py <action> [--参数名 参数值]...
```

**重要说明：**
- `<action>` 是动作名称，必须是以下支持的动作之一
- 参数使用 `--参数名 参数值` 格式传递，无需 JSON
- 必需参数必须提供，可选参数可不传
- 所有命令返回 JSON 格式结果

### 3.13 check_new_messages - 检查新消息

**功能：** 检查是否有新消息（单次检测，支持跨进程状态持久化）

**调用：**
```bash
python {baseDir}/agent.py check_new_messages
```

**参数说明：** 无参数

**返回值：**

有新消息：
```json
{
  "status": "success",
  "data": {
    "has_new_messages": true,
    "new_messages": [
      {
        "contact": "张三",
        "message": "今天下午开会",
        "position": {"x": 140, "y": 150},
        "is_from_me": false
      }
    ],
    "all_contacts": [...],
    "total_new": 1
  }
}
```

无新消息：
```json
{
  "status": "success",
  "data": {
    "has_new_messages": false,
    "new_messages": [],
    "all_contacts": [...],
    "total_new": 0
  }
}
```

**用途：**
- 单次检测，由用户控制调用时机
- 检测到新消息后，使用 `auto_reply` 回复
- **支持跨进程状态持久化**：联系人状态保存在 `wechat_contact_states.json` 文件中，即使多次调用也能正确检测新消息
- **支持手动回复检测**：自动检测用户是否已手动回复，避免重复回复

**持久化机制：**
- 首次调用：所有联系人标记为"已回复"状态
- 后续调用：对比上次保存的消息，检测变化
- 状态文件：`wechat_contact_states.json`（保存在 agent.py 同目录）

**手动回复检测机制：**
- 检查聊天列表中最后一条消息的发送者
- 如果最后一条消息来自"我"（`is_from_me=True`），说明已手动回复
- 自动将状态标记为"已回复"，避免重复自动回复
- 即使之前已标记为"未回复"，也能检测到手动回复并更新状态

**使用流程示例：**
```
用户: 看看有没有新消息
模型: [调用 check_new_messages()] 有1条新消息，张三说"今天下午开会"

用户: 回复他"好的"
模型: [调用 auto_reply --name "张三" --message "好的"] 已回复张三

用户: 再看看有没有新消息
模型: [调用 check_new_messages()] 暂无新消息
```

## 5. 常见操作流程

### 5.1 查看聊天记录流程

**场景：用户说"查看和XXX的聊天记录"、"看看和XXX的聊天"**

**标准流程：**
```
步骤1: 获取聊天列表
  → 调用 get_chat_list() 获取联系人列表

步骤2: 查找目标联系人
  → 在返回的 contacts 中查找名称匹配的联系人
  → 获取其 position 坐标

步骤3: 点击联系人进入聊天窗口
  → 调用 click_contact --name "XXX" 或使用坐标点击
  → ⚠️ 注意：点击后需要验证是否成功进入聊天窗口

步骤4: 验证聊天窗口状态（重要！）
  → 点击联系人后，再次调用 get_ocr_result() 或 screenshot()
  → 检查右侧是否出现聊天窗口（通常包含聊天记录区域）
  → 如果右侧没有聊天窗口，说明可能已经在该聊天窗口，点击反而关闭了
  → 如果确认聊天窗口被关闭，再次点击联系人重新打开

步骤5: 查看聊天记录
  → 可通过 get_ocr_result() 识别聊天内容
  → 或通过 scroll() 滚动查看更多历史记录
```

**示例对话：**
```
用户: 查看和张三的聊天记录
模型: 
  1. [调用 get_chat_list()] 找到张三
  2. [调用 click_contact --name "张三"] 点击张三
  3. [调用 get_ocr_result()] 验证聊天窗口状态
  4. 如果窗口已打开，返回聊天内容；如果窗口被关闭，重新点击打开
  5. [可选] [调用 scroll --direction up] 查看更多历史记录
```

### 5.2 发送消息流程

**场景：用户说"给XXX发消息"、"和XXX说话"、"告诉XXX"**

**⚠️ 核心问题：微信联系人列表的开关切换行为**
- 如果当前**不在**XXX的聊天窗口，点击联系人会**打开**聊天窗口
- 如果当前**已经在**XXX的聊天窗口，点击联系人会**关闭**聊天窗口
- **因此，和某个人说话时，必须判断当前是否已经在他的聊天窗口！**

**标准流程：**
```
步骤1: 获取当前界面状态
  → 调用 get_ocr_result() 或 screenshot() 查看当前界面
  
步骤2: 判断当前是否已在目标聊天窗口
  → 检查右侧聊天窗口的标题或内容
  → 查看窗口标题是否显示"XXX"
  → 查看右侧是否有聊天记录区域
  
步骤3: 根据判断结果执行不同操作
  
  情况A - 已经在XXX的聊天窗口：
    → 直接跳到步骤5发送消息
    → ⚠️ 不要点击联系人，否则会关闭窗口！
  
  情况B - 不在XXX的聊天窗口：
    → 执行步骤4，点击联系人进入聊天窗口

步骤4: 点击联系人并验证窗口状态（关键！）
  → 调用 click_contact --name "XXX"
  → ⚠️ 点击后立即验证：调用 get_ocr_result() 检查右侧聊天窗口
  → 检查点：
    - 右侧是否出现聊天记录区域
    - 窗口标题是否显示"XXX"
    - 底部是否有输入框
  → 如果右侧没有聊天窗口：
    - 说明之前已经在该窗口，点击导致关闭了
    - 需要再次调用 click_contact --name "XXX" 重新打开
    - 再次验证直到确认窗口打开

步骤5: 确认窗口打开后发送消息
  → 调用 send_message --message "内容"
```

**示例对话：**
```
用户: 和李四说"你好"
模型:
  1. [调用 get_ocr_result()] 检查当前界面
  
  2. 分析当前状态：
     - 如果右侧聊天窗口标题是"李四" → 说明已在李四的聊天窗口
     - 如果右侧是其他联系人或空白 → 说明不在李四的聊天窗口
  
  3. 情况A - 已在李四的聊天窗口：
     [调用 send_message --message "你好"] 直接发送
     
  4. 情况B - 不在李四的聊天窗口：
     a. [调用 click_contact --name "李四"] 点击李四
     b. [调用 get_ocr_result()] 验证窗口状态
     c. 如果窗口未打开（右侧空白）：
        [再次调用 click_contact --name "李四"] 重新点击
        [再次验证]
     d. 确认窗口打开后：
        [调用 send_message --message "你好"] 发送消息
```

## 6. 重要机制说明

### 6.1 点击联系人的窗口状态判断机制（核心！）

**问题背景：**
- 微信的联系人列表有**开关切换行为**
- 如果当前**不在**某人的聊天窗口，点击联系人会**打开**聊天窗口
- 如果当前**已经在**某人的聊天窗口，再次点击联系人会**关闭/隐藏**聊天窗口

**正确的处理逻辑：**
```
1. 点击联系人后，必须验证聊天窗口状态
   → 点击后立即通过 OCR 检查右下角是否有"发送"按钮
   → 发送按钮是聊天窗口的核心特征，出现则表示窗口已打开

2. 判断结果：
   - 如果右下角出现"发送"按钮 → 成功打开，可以继续操作
   - 如果没有"发送"按钮 → 说明之前已在该窗口，点击导致关闭
     → 需要再次点击联系人重新打开

3. 自动化实现：
   click_contact 已内置验证机制，会自动处理窗口状态判断
   - 点击联系人后自动验证
   - 如果窗口未打开，自动重新点击
   - 最多尝试2次
```

**状态判断的视觉特征：**
```
聊天窗口已打开的特征：
- 右下角有"发送(S)"或"发送"按钮（核心判断依据）
- 右侧有聊天记录区域（包含消息气泡）
- 窗口标题显示联系人名称

聊天窗口未打开的特征：
- 右下角没有"发送"按钮
- 右侧为空白或显示"选择聊天"提示
- 或者右侧显示之前其他联系人的聊天内容
```

### 6.3 定时监控消息的正确使用方式

**⚠️ 重要：模型如果需要实现定时监控（如每10分钟检查一次），必须遵循以下流程**

**场景：模型实现定时监控微信消息**

**第一次启动监控时：**
```
步骤1: 重置状态（清除旧的状态文件）
  → 调用 reset_contact_states()
  
步骤2: 初始化当前状态
  → 调用 check_new_messages()
  → 此时会将当前所有联系人标记为"已回复"状态
  → 状态保存到 wechat_contact_states.json 文件
  
步骤3: 返回结果
  → 第一次调用 has_new_messages 通常为 false（因为没有新消息）
  → 这一步是为了建立初始状态基线
```

**后续定时检查（每隔N分钟）：**
```
步骤1: 直接调用 check_new_messages()
  → ⚠️ 不要再调用 reset_contact_states()！
  → 系统会自动从文件加载上次保存的状态
  
步骤2: 系统自动检测变化
  → 对比当前消息与上次保存的消息
  → 如果有变化且是对方发的消息 → has_new_messages = true
  → 如果消息来自我（手动回复） → 自动标记为已回复
  
步骤3: 根据结果处理
  → 有新消息 → 提示用户或自动回复
  → 无新消息 → 继续等待下一次定时检查
```

**关键要点：**
```
✅ 正确做法：
  - 启动时：reset_contact_states() → check_new_messages()
  - 后续：只调用 check_new_messages()，不重置

❌ 错误做法：
  - 每次定时都调用 reset_contact_states()
  - 这会导致每次都是"初始状态"，检测不到新消息变化
```

**示例：定时监控的完整流程**
```
用户: 帮我每隔10分钟检查一次微信新消息

模型:
  【第一次 - 启动监控】
  1. [调用 reset_contact_states()] 重置状态
     → ⚠️ 必须检查返回值 status: "success"
  2. [调用 check_new_messages()] 初始化状态基线
     → ⚠️ 必须检查返回值 status: "success"
     → 确认返回了 all_contacts 列表
  3. 告知用户：监控已启动，当前无新消息，10分钟后再次检查

  【10分钟后 - 第一次定时检查】
  1. [调用 check_new_messages()] 检查新消息
  2. 如果 has_new_messages=true：
     a. 告知用户"XXX发来新消息：XXX"
     b. [调用 auto_reply --name "联系人" --message "回复内容"] 自动回复
     c. 回复内容可以是：
        - 用户预设的回复模板
        - 根据消息内容智能生成
        - 默认回复"收到，稍后回复"
  3. 如果 has_new_messages=false → 继续等待

  【20分钟后 - 第二次定时检查】
  1. [调用 check_new_messages()] 检查新消息
  2. 同上...

  【注意】后续所有检查都只调用 check_new_messages()，不再重置状态
```

**⚠️ 自动回复规则（重要！）**

模型检测到新消息后，**必须自动回复**，遵循以下规则：

```
【自动回复触发条件】
check_new_messages() 返回 has_new_messages: true
  → 遍历 new_messages 列表
  → 对每条新消息调用 auto_reply

【回复内容来源】
1. 优先使用用户预设的回复模板（如果有）
2. 根据消息内容智能生成回复
3. 默认回复："收到，稍后回复您"

【自动回复流程】
检测到新消息 → 告知用户 → 自动调用 auto_reply → 告知用户已回复
```

**示例对话：**
```
【定时检查发现新消息】
模型: 检测到"小许"发来新消息："今天下午开会"
      正在自动回复...
      [调用 auto_reply --name "小许" --message "收到，稍后回复您"]
      已自动回复"小许"

【多条新消息】
模型: 检测到2条新消息：
      - 小许："今天下午开会"
      - 张三："文件收到了吗"
      正在逐一自动回复...
      [调用 auto_reply --name "小许" --message "收到，稍后回复您"]
      [调用 auto_reply --name "张三" --message "收到，稍后回复您"]
      已全部回复完成
```

**⚠️ 重要：定时任务的自我检查机制**

模型设置定时任务后，必须进行自我检查，确保定时任务真正成功运行：

```
【必须检查的事项】
1. 初始化步骤必须有返回值
   → reset_contact_states() 必须返回 {"status": "success", ...}
   → check_new_messages() 必须返回 {"status": "success", "data": {...}}
   → 如果返回 status: "error" → 定时任务失败，必须告知用户并重新尝试

2. 初始化返回值必须包含有效数据
   → check_new_messages() 返回的 all_contacts 必须有内容
   → 如果 all_contacts 为空数组 → 可能微信未打开或无法识别，定时任务无意义

3. 定时任务必须能持续执行
   → 第一次检查成功后，确认定时器已设置
   → 后续每次检查都要验证返回值
   → 如果某次检查返回 error → 记录错误并告知用户

【错误处理示例】
```
情况1: 初始化失败
  → reset_contact_states() 返回 {"status": "error", "message": "微信未运行"}
  → 模型应告知用户："定时任务启动失败，请确保微信已打开"
  → 不要继续设置定时器

情况2: 检查失败
  → check_new_messages() 返回 {"status": "error", ...}
  → 模型应记录错误，告知用户："本次检查失败，将在下次定时继续尝试"
  → 定时任务继续运行，不中断

情况3: 返回数据为空
  → check_new_messages() 返回 all_contacts: []
  → 可能是微信窗口未激活或 OCR 识别失败
  → 告知用户："无法获取联系人列表，请检查微信状态"
```

### 6.4 手动回复检测机制

**问题背景：**
用户可能手动在微信中回复消息，系统需要识别这种情况，避免重复自动回复。

**检测原理：**
```
每次调用 check_new_messages 时：
1. 获取聊天列表，检查每个联系人的最后一条消息
2. 如果最后一条消息来自"我"（is_from_me=True）：
   → 说明用户已手动回复
   → 将状态标记为"已回复"（replied=True）
   → 不加入新消息列表
3. 如果最后一条消息来自对方（is_from_me=False）：
   → 检查是否已标记为"已回复"
   → 如果未回复，加入新消息列表
```

**示例场景：**
```
时间线：
10:00 - 张三发消息"你好"
10:05 - check_new_messages 检测到新消息，replied=False
10:10 - 用户手动回复"好的"
10:15 - check_new_messages 检测到：
        - 最后一条消息是"好的"（来自我）
        - 自动标记 replied=True
        - 不再提示需要回复
```

**优势：**
- ✅ 自动检测手动回复，无需额外操作
- ✅ 避免重复回复尴尬
- ✅ 与自动回复无缝配合
