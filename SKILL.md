# 微信自动化技能

## 1. Description（技能详细说明）

这是一个微信客户端自动化控制工具，可以唤醒微信窗口、截图、识别界面内容、模拟点击和输入操作、发送消息、检查新消息。

**注意：微信需要用户手动打开，工具不会自动启动微信。**

**核心特性：**
- 自动唤醒微信窗口（Ctrl+Alt+W 快捷键，仅当微信在托盘区时）
- 所有操作前自动激活窗口，确保窗口在前台
- 每次操作都获取最新截图，不使用缓存
- 支持多窗口标题匹配（"微信"、"WeChat"、"Weixin"）
- **OCR结果由AI解析，不再在代码中固定解析，适应不同窗口大小**
- **新消息检测：依次点击前5个联系人，查看右侧聊天记录，准确区分消息来源**

**⚠️ 跳过的联系人类型：**
- 文件传输助手、File Transfer、文件传输
- 公众号、服务号、微信支付
- 腾讯、系统、官方

## 2. When to use（触发场景）

当用户说以下内容时，使用此技能：

- "打开微信"、"唤醒微信"、"显示微信"
- "帮我截个微信的图"、"微信截图"
- "微信界面上有什么内容"、"识别微信界面文字"
- "点击微信上的XXX"、"在微信里输入XXX"
- "发送消息"、"在当前聊天发送XXX"
- "滚动微信聊天记录"
- "帮我操作微信"相关任务
- "微信状态"、"微信是否在运行"
- **"看看谁给我发消息了"、"查看聊天列表"、"检查新消息"** → 使用 `get_chat_list` 或 `check_new_messages`
- **"查看和XXX的聊天记录"、"看看和XXX的聊天"、"读取XXX的聊天内容"** → 使用完整流程（见第5.1节）
- **"给XXX发消息"、"和XXX说话"、"告诉XXX"** → 使用完整流程（见第5.2节）
- **"回复XXX"** → 使用 `auto_reply`


- `<action>` 是动作名称，必须是以下支持的动作之一

### 3.10 get_chat_list - 获取聊天列表（OCR原始结果）

**功能：** 获取微信窗口的OCR识别原始结果，由AI自行分析联系人和消息

**调用：**
```bash
python {baseDir}/agent.py get_chat_list
```

**参数说明：**

|| 参数名 | 类型 | 必需 | 默认值 | 说明 |
||--------|------|------|--------|------|
|| --count | int | 否 | 5 | 保留参数（兼容旧接口） |

**返回值：**

成功：
```json
{
  "status": "success",
  "data": {
    "ocr_results": [
      {
        "text": "识别的文本",
        "confidence": 0.95,
        "position": {"x": 100, "y": 200},
        "box": [[x1,y1], [x2,y2], [x3,y3], [x4,y4]]
      }
    ],
    "window_rect": {"left": 0, "top": 0, "width": 800, "height": 600},
    "total": 10
  }
}
```

**重要说明：**
- 不再在代码中解析联系人和消息，直接返回原始OCR结果
- AI需要根据坐标位置自行判断哪些是联系人、哪些是消息内容
- `position` 是左上角坐标，`box` 是四个角的坐标

**AI分析OCR结果的规则：**

```
【微信窗口布局分析】

1. 左侧聊天列表区域
   - 使用固定宽度 + 动态调整策略：
     * 小窗口（<700px）：使用 25% 比例
     * 中等窗口（700-900px）：使用固定 250px
     * 大窗口（>900px）：使用固定 250px，不超过 35%
   - 联系人名称：通常在X坐标 50-200 范围
   - 消息预览：在联系人名称下方或右侧
   - 时间：X坐标较大（通常在右侧）
   - 未读数量：可能显示为红色数字

2. 右侧聊天窗口区域（X坐标较大）
   - 聊天窗口标题：显示当前聊天对象名称（在顶部）
   - 聊天记录：多条消息按时间排列
   - 消息气泡：我发的在右侧，对方发的在左侧
   - 输入框：底部区域

【关键判断逻辑 - 重要！】

1. ⚠️ 跳过的联系人类型（不需要回复）
   以下类型的联系人直接跳过，不处理：
   - 文件传输助手、File Transfer、文件传输
   - 公众号、服务号、微信支付
   - 腾讯、系统、官方
   - 包含以上关键词的联系人名称

2. 区分左侧聊天列表和右侧聊天窗口
   - 左侧边界使用固定宽度 + 动态调整：
     * 小窗口（<700px）：window_width * 0.25
     * 中等/大窗口（>=700px）：固定 250px
   - X < 左侧边界 → 左侧列表
   - X > 左侧边界 → 右侧聊天窗口

3. ⚠️ 区分"我发的消息"和"对方发的消息"（核心！最容易出错！）

   【左侧聊天列表 - 消息预览判断】
   - 如果消息预览以"我:"开头 → 我发的最后一条消息，忽略此联系人
   - 如果消息预览是"[语音]"、"[图片]"等且不以"我:"开头 → 需要进入聊天窗口确认
   - 如果消息预览是普通文本且不以"我:"开头 → 对方发的，需要回复

   【右侧聊天窗口 - 气泡位置判断】
   - 消息气泡 X 坐标 > window_width * 0.6 → 我发的消息（右侧）
   - 消息气泡 X 坐标 < window_width * 0.5 → 对方发的消息（左侧）
   - 中间区域需要根据上下文判断

   【关键规则 - 避免回复自己】
   - 只有"对方发的消息"才需要回复
   - 如果判断是"我发的消息"，绝对不要回复！
   - 不确定时，默认假设是"我发的消息"，不回复

4. 右侧聊天窗口的消息归属判断
   - 聊天窗口顶部显示当前聊天对象名称
   - 该窗口内的所有消息都属于这个联系人
   - ⚠️ 注意：右侧可能显示多条聊天记录，需要根据Y坐标区分不同消息

5. 时间识别
   - 格式：HH:MM、昨天、前天、日期等
   - 通常X坐标较大，在消息右侧或列表右侧

6. 聊天记录匹配（重要！）
   - 右侧聊天窗口可能显示多条聊天记录
   - 需要根据Y坐标区分不同消息
   - 同一联系人的消息按Y坐标从小到大排列（时间从早到晚）
```

**用途：**
- AI分析OCR结果，识别联系人列表
- AI判断哪些联系人有新消息
- AI计算点击坐标，用于后续操作

## 5. 核心流程：定时监控新消息

### 5.1 整体思路

```
核心流程：
1. 调用 check_new_messages() 获取当前联系人列表（已解析出名称、预览、时间）
2. 对比 saved_records 中保存的预览消息
3. 发现新消息 → 点击联系人 → 验证窗口 → 回复 → 更新预览
4. 新联系人（本地无记录）且有预览消息 → 当做新消息处理
```

**关键改进：**
- 不再需要依次点击每个联系人查看聊天记录
- 直接从联系人列表的预览消息判断是否有新消息
- 预览消息可能被截断，需要回复完整聊天内容时才点击进入对话框

### 5.2 初始化流程（第一次启动监控）

```
步骤1: 重置聊天记录
  → 调用 reset_chat_records()

步骤2: 获取当前联系人列表
  → 调用 check_new_messages()
  → 返回 contacts（已解析的联系人列表，包含 name, preview, time, screen_x, screen_y）
  → is_first_init = true

步骤3: 保存所有联系人的预览消息
  对每个联系人：
    a. 调用 save_chat_record --name "联系人名" --message "预览消息" --time "时间"
    b. 不需要点击进入对话框（预览消息已足够）

步骤4: 更新联系人顺序（可选）
  → 调用 update_contacts_order --order '["张三", "李四", ...]'

步骤5: 设置定时任务
```

### 5.3 定时检查流程

```
步骤1: 获取当前联系人列表
  → 调用 check_new_messages()
  → 返回 contacts（当前联系人列表）和 saved_records（上次保存的记录）

步骤2: 对比预览消息找出新消息
  对每个联系人：
    a. 检查联系人名称是否在 saved_records["contacts"] 中
    b. 如果不存在 → 新联系人，当做新消息处理
    c. 如果存在 → 对比 preview 和 saved_records["contacts"][name]["last_opponent_message"]
    d. 如果 preview 不同 → 有新消息

步骤3: 处理新消息
  对每个有新消息的联系人：
    a. 调用 click_contact --x screen_x --y screen_y 进入对话框
    b. 调用 verify_chat_window --expected_name "联系人名" 验证窗口
    c. （可选）获取完整聊天记录判断是否需要回复
    d. 调用 send_message --message "回复内容" 发送回复
    e. ⚠️ 立即调用 update_preview_after_reply --name "联系人名" 更新预览
       （防止自己的回复变成预览导致下次误判）

步骤4: 设置下一次定时任务
```

### 5.4 区分消息来源的关键规则

```
【右侧聊天窗口 - 气泡位置判断】

1. 获取窗口宽度 window_width
2. 对每条消息计算X坐标中心点：
   center_x = (box[0][0] + box[2][0]) / 2

3. 判断消息来源：
   - center_x > window_width * 0.6 → 我发的消息（右侧气泡）
   - center_x < window_width * 0.5 → 对方发的消息（左侧气泡）
   - 中间区域需要根据上下文判断

4. 找对方最后一条消息：
   - 过滤出 sender="对方" 的消息
   - 按Y坐标排序，取最后一条（Y最大的）

【关键规则】
- 只有"对方发的消息"才需要回复
- 如果判断是"我发的消息"，绝对不要回复！
- 不确定时，默认假设是"我发的消息"，不回复
```

## 6. 重要机制说明

### 6.1 OCR结果解析机制（核心！）

**设计理念：**
- 微信窗口大小可能变化，固定坐标解析不可靠
- OCR返回原始结果，由AI智能分析
- AI根据相对位置关系判断界面元素

**AI解析OCR的关键规则：**

```
【区域划分】

窗口宽度 = window_rect.width
左侧边界计算（适应不同窗口大小）：
- 小窗口（<700px）：分界线X = window_width * 0.25
- 中等窗口（700-900px）：分界线X = 250
- 大窗口（>900px）：分界线X = 250（不超过 window_width * 0.35）

左侧聊天列表：X < 分界线
右侧聊天窗口：X > 分界线

【左侧列表解析】

1. 按Y坐标分组
   - Y坐标相近（差值 < 40像素）的文本属于同一联系人
   - 每组从上到下排列

2. 每组内的元素
   - 名称：X坐标最小的主要文本
   - 消息预览：名称下方的文本
   - 时间：X坐标最大的文本

3. 过滤规则
   - "搜索" 等UI元素应忽略
   - 纯数字可能是未读数量或时间

【右侧聊天窗口解析】

1. 窗口标题
   - 位于右侧顶部区域
   - 显示当前聊天对象名称

2. 聊天消息
   - 按Y坐标从上到下排列
   - 我发的消息：X坐标较大（右侧）
   - 对方发的消息：X坐标较小（左侧）

3. 群聊特殊处理
   - 每条消息前可能有发送者名称
   - 格式："发送者名称: 消息内容" 或 "发送者名称 消息内容"

【点击坐标计算】

联系人点击坐标：
- X = window_rect.left + 列表宽度 * 0.5
- Y = window_rect.top + 联系人Y中心 + 顶部偏移

列表宽度 ≈ window_width * 0.28
顶部偏移 ≈ window_height * 0.08
```

### 6.2 定时监控消息的正确使用方式

**⚠️ 极其重要：定时任务触发时必须先初始化状态！**

**使用 cron.add 工具设置定时任务：**

```
【第一次启动监控时】必须初始化！

步骤1: 重置状态
  → 调用 reset_chat_records()

步骤2: 获取当前联系人列表
  → 调用 check_new_messages()
  → 返回 contacts（已解析的联系人列表）

步骤3: 保存所有联系人的预览消息
  对每个联系人：
    → 调用 save_chat_record --name "联系人名" --message "预览消息" --time "时间"

步骤4: 设置定时任务
  → 使用 cron.add 工具设置下次检查时间
  → schedule.at = 当前时间 + 间隔时间（ISO 8601 格式）
  → payload.kind = "agentTurn"
  → payload.message = "检查微信新消息"

【定时任务触发时】对比消息变化！

步骤1: 获取当前联系人列表
  → 调用 check_new_messages()
  → 返回 contacts 和 saved_records

步骤2: ⚠️ 过滤不需要处理的联系人

  跳过以下类型的联系人（不需要回复）：
  - 文件传输助手、File Transfer、文件传输
  - 公众号、服务号、微信支付
  - 腾讯、系统、官方
  - 折叠聊天、折叠的聊天
  - 包含以上关键词的联系人名称

步骤3: 对比预览消息

  对每个联系人：
    a. 检查联系人名称是否在 saved_records["contacts"] 中
    b. 如果不存在 → 新联系人，当做新消息处理
    c. 如果存在 → 对比 preview 和 saved_records["contacts"][name]["last_opponent_message"]
    d. 如果 preview 不同 → 有新消息

步骤4: 批量处理新消息
  ⚠️ 重要：必须一次性处理所有新消息，中间不要再次获取OCR！

  → 在 AI 内存中记录所有【有新消息】的联系人名称和坐标
  → 按从上到下的顺序逐一处理：
    - auto_reply --x screen_x --y screen_y --message "回复内容"
    - update_preview_after_reply --name "联系人名"

步骤5: 设置下一次定时任务
```

**⚠️ 关于新联系人的处理：**

```
【新联系人判断】
如果联系人名称不在 saved_records["contacts"] 中：
- 这是新出现在列表中的联系人
- 通常是因为对方发来了新消息
- 应该当做新消息处理

【示例】
假设 saved_records 中有：张三、李四、王五
当前列表出现：张三、李四、赵六（赵六是新联系人）
→ 赵六不在保存记录中 → 当做新消息处理
```

**⚠️ 为什么不能中间获取 OCR？**

```
错误做法：
1. OCR 获取状态，发现 A、B、C 有新消息
2. 点击 A 回复 → 右侧窗口变为 A 的聊天
3. 再次 OCR 获取状态 → ❌ 错误！
   - 右侧窗口已经不是初始状态
   - 左侧列表可能已经滚动变化
   - 之前计算的 B、C 坐标可能失效

正确做法：
1. OCR 获取状态，发现 A、B、C 有新消息
2. 在 AI 内存中记录：A(坐标1), B(坐标2), C(坐标3)
3. 直接用记录的坐标：
   - auto_reply --x 坐标1 --y 坐标1 --message "回复A"
   - update_preview_after_reply --name "A"  ← 回复后立即更新预览
   - auto_reply --x 坐标2 --y 坐标2 --message "回复B"
   - update_preview_after_reply --name "B"  ← 回复后立即更新预览
   - auto_reply --x 坐标3 --y 坐标3 --message "回复C"
   - update_preview_after_reply --name "C"  ← 回复后立即更新预览
```

**⚠️ 点击坐标已自动计算**

```
check_new_messages() 返回的每个联系人已包含：
- screen_x: 屏幕绝对X坐标（可直接用于点击）
- screen_y: 屏幕绝对Y坐标（可直接用于点击）

使用方法：
click_contact --x contact["screen_x"] --y contact["screen_y"]
auto_reply --x contact["screen_x"] --y contact["screen_y"] --message "回复内容"
```

### 6.3 自动回复规则

**模型检测到新消息后，必须自动回复，遵循以下规则：**

```
【回复内容生成规则】

1. 能办到的事情 → 执行并回复结果
   - 查询信息（天气、价格、新闻等）→ 调用相应工具查询后回复结果
   - 简单计算 → 计算后回复答案
   - 设置提醒 → 设置后确认

2. 办不到的事情 → 明确告知原因
   - 需要物理操作 → "抱歉，我无法进行物理操作"
   - 超出AI能力范围 → "抱歉，这个我暂时做不到"

3. 闲聊/问候 → 自然回复
   - "你好" → "你好，有什么可以帮你的吗？"
   - "在吗" → "在的，请说"

4. 不明确的消息 → 礼貌询问
   - "您说的我不太理解，能详细说明一下吗？"
```

## 7. 注意事项

1. **坐标系统**：所有坐标均为屏幕绝对坐标，左上角为 (0, 0)
2. **OCR解析**：由AI根据相对位置智能分析，不再固定解析
3. **窗口大小**：适应不同窗口大小，通过相对比例判断位置
4. **消息发送**：`send_message` 会自动定位输入框，无需手动点击
5. **窗口前置**：所有操作都会自动将微信窗口置顶
6. **无缓存**：每次 OCR 和截图都获取最新画面
7. **消息来源判断**：通过右侧聊天窗口的气泡位置判断，X > 60% 是我发的，X < 50% 是对方发的
8. **聊天记录持久化**：聊天记录保存在 `wechat_chat_records.json` 文件中
9. **定时任务初始化**：设置定时任务时必须先调用 `reset_chat_records` 初始化
10. **批量处理**：check_new_messages 返回所有联系人和坐标，批量处理新消息，中间不要再获取 OCR
11. **中文编码**：脚本输出使用 GBK 编码，适配 Windows CMD 环境
12. **跳过联系人**：文件传输助手、公众号、微信支付等不需要回复
13. **回复后更新预览**：⚠️ 回复消息后必须调用 `update_preview_after_reply` 更新预览，防止自己的回复变成预览导致下次误判为新消息
